# 使用示例

下面的示例可以直接复制到 NeoCode 会话里。把 `{...}` 替换成你项目里的真实路径、日志或需求描述。

## 场景 1：读懂陌生项目

```text
请先阅读当前项目的目录结构，按模块说明每个目录负责什么，并指出我应该从哪些文件开始了解主流程。
```

适合第一次打开项目时使用。文件读取和搜索通常不需要额外确认。

## 场景 2：先读项目规则

```text
请先阅读当前工作区可见的 AGENTS.md 和 README，概括本项目要求你遵守的协作规则。
```

如果项目已经维护了 `AGENTS.md`，这个提示能帮助你确认 Agent 理解了项目规则。

## 场景 3：先规划，再修改

```text
我想添加这个功能：{feature description}。
请先阅读相关代码，给出最小实现方案、需要修改的文件和建议验证命令。暂时不要改文件。
```

确认方案后再说：

```text
请按刚才的方案实现，保持现有代码风格，并补充必要测试。
```

这种方式适合影响范围不确定的功能或修复。

## 场景 4：定位并修复 bug

```text
运行测试时看到以下错误，请定位根因并给出修复方案：

{test log}
```

确认后继续：

```text
请按刚才的方案修改代码，并运行最小相关测试。如果测试失败，请分析原因并继续修复。
```

遇到删除、重置或大范围改写时，建议保持权限为 Ask，先看清操作内容。

## 场景 5：补测试

```text
请为 {source file} 里的 {function name} 补单元测试，覆盖正常路径、空输入和错误返回。测试文件放在 {test file}。
```

验证：

```text
请运行相关测试。如果失败，请分析原因并修复。
```

## 场景 6：保存长期偏好

```text
/remember 我习惯先看测试再改代码
```

之后你可以查看：

```text
/memo
```

记忆适合长期偏好，不适合密钥、Token 或一次性任务要求。

## 场景 7：启用 Skill

```text
/skills
/skill use go-review
```

然后发起任务：

```text
请审查当前改动，优先列出行为回归、边界条件和测试缺口。
```

Skill 只影响当前会话的工作方式，不会新增工具或跳过权限确认。

## 场景 8：长会话压缩后继续

```text
/compact
```

压缩后补一句当前目标：

```text
继续刚才的修复，当前目标是让 go test ./internal/config 通过，只改必要文件。
```
